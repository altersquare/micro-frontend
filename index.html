What Are Micro Frontends?

Micro Frontends are an architectural pattern that extends the microservices concept to frontend development. Instead of building a monolithic frontend application, you break it down into smaller, independently deployable applications that work together to create a cohesive user experience.

Why Module Federation is Leading the Way

Module Federation has emerged as the most popular approach because it offers:
  • Runtime Flexibility: Load modules on demand without rebuild
  • Optimized Bundle Size: Share dependencies intelligently
  • Independent Deployment: True deployment independence
  • Framework Agnostic: Works with React, Vue, Angular, and more
  • Strong Ecosystem: Growing community and tooling support
// Example: Module Federation Configuration
// Host Application (Shell)
const ModuleFederationPlugin = require('@module-federation/webpack');

module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'shell',
      remotes: {
        'commerce-app': 'commerceApp@http://localhost:3001/remoteEntry.js',
        'user-app': 'userApp@http://localhost:3002/remoteEntry.js',
      },
      shared: {
        react: { singleton: true },
        'react-dom': { singleton: true }
      }
    })
  ]
}

// Remote Application (Commerce)
module.exports = {
  plugins: [
    new ModuleFederationPlugin({
      name: 'commerceApp',
      exposes: {
        './ProductCatalog': './src/components/ProductCatalog',
        './ShoppingCart': './src/components/ShoppingCart'
      },
      shared: {
        react: { singleton: true },
        'react-dom': { singleton: true }
      }
    })
  ]
}

export const counterInterface = {
  isReady() { return true },
  getValue() { return useCommonStore().num },
  increment() { useCommonStore().increment() },
  decrement() { useCommonStore().decrement() },

// Consumer with activity log (demo-counter-app/src/components/CounterDemo.vue)
import { onMounted, onUnmounted, ref } from 'vue'
const currentValue = ref(0)
const log = ref([])
let unsubscribe = null

onMounted(async () => {
  try {
    const { counterInterface } = await import('shellApp/interfaces')
    if (!counterInterface.isReady()) return
    unsubscribe = counterInterface.subscribe((val, mutation) => {
      currentValue.value = val
      log.value.unshift({ ts: Date.now(), val, mutation })
    })
    currentValue.value = counterInterface.getValue()
  } catch (e) {
    console.error('Counter interface unavailable', e)
  }
})

onUnmounted(() => { if (unsubscribe) { unsubscribe(); unsubscribe = null } })

Best Practices & Production Guidelines

üèóÔ∏è

Architecture & Design

Foundations to keep teams decoupled and software evolvable
  • Organize by business domain (not tech layers)
  • Minimize cross-MFE dependencies; use narrow contracts
  • Define clear ownership and SLAs per module
  • Plan fault tolerance and graceful degradation paths
üß©

Module Federation

Configuration patterns that scale across environments
  • Use env-based remote URLs for staging/prod
  • Share frameworks as singletons; allow lib poly-versioning
  • Version and validate exposed contracts
  • Guard remote failures with retries and fallbacks
üé®

UI/UX Consistency

One experience, many teams
  • Adopt a shared design system as a federated package
  • Publish tokens (color, spacing, type) as source of truth
  • Automate visual regression and accessibility checks
  • Establish review gates for shared components
‚ö°

Performance & Delivery

Fast by default, faster in practice
  • Analyze bundles frequently; dedupe shared deps
  • Lazy-load MFEs and prefetch likely next routes
  • Leverage CDN caching and immutable asset naming
  • Track Core Web Vitals per module in prod

// Vite: Module Federation best practices (RSPack/Module Federation plugin for Vite)
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'
import federation from '@module-federation/vite'

export default defineConfig({
  plugins: [
    react(),
    federation({
      name: 'shell',
      filename: 'remoteEntry.js',
      remotes: {
        // Env-driven remotes for each environment
        drawing: `${process.env.VITE_DRAWING_URL}/remoteEntry.js`,
        shapes: `${process.env.VITE_SHAPES_URL}/remoteEntry.js`
      },
      exposes: {
        // Keep API surfaces small and stable
        './canvasApi': './src/canvas/api.ts'
      },
      shared: {
        // Singletons for frameworks; others can be poly-versioned
        react: { singleton: true, requiredVersion: '^18.0.0' },
        'react-dom': { singleton: true, requiredVersion: '^18.0.0' }
      }
    })
  ],
  define: {
    // Ensure env vars are injected at build time
    'process.env.VITE_DRAWING_URL': JSON.stringify(process.env.VITE_DRAWING_URL),
    'process.env.VITE_SHAPES_URL': JSON.stringify(process.env.VITE_SHAPES_URL)
  }
})

Decision Framework: When to Use Micro Frontends

Perfect Candidates for Micro Frontends

Use Micro Frontends When:
  • Large Teams (8+ developers): Multiple teams working on the same frontend
  • Complex Applications (50,000+ lines): Application has grown beyond manageable size
  • Independent Features: Clear business domain boundaries exist
  • Different Technologies: Teams want to use different frameworks or libraries
  • Rapid Development: Need to deploy features independently
  • Legacy Migration: Gradually modernizing a monolithic application
  • Multi-Tenant Applications: Different features for different customer segments
  • Organizational Autonomy: Teams want full ownership of their features

When NOT to Use Micro Frontends

Avoid Micro Frontends When:
  • Small Applications: Simple apps with limited complexity
  • Small Teams (< 8 developers): Overhead exceeds benefits
  • Tight Coupling: Features are heavily interdependent
  • Performance Critical: Milliseconds matter more than development speed
  • Limited Resources: Lack expertise or time for complex setup
  • Prototypes/MVPs: Need to move fast with simple architecture
  • Short-term Projects: Not worth the architectural investment
